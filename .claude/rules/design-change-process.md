# 設計変更時のプロセス

アーキテクチャ・データモデル・処理フローなど「設計」に関わる変更を行う場合は、次の順序を必ず守る。

1. **設計書を修正または作成する** — 既存の設計書（`~/.claude/plans/` 配下のplanファイル、または `CLAUDE.md` のアーキテクチャ節）を更新する。設計書がまだ無い変更なら新規に作成する。設計書には変更の背景（なぜ必要か）と変更後の構造を書く。
2. **タスクを提示する** — 設計書の内容をもとに、具体的な実装タスク（変更するファイル・関数・データモデル）を一覧化してユーザーに提示する。
3. **コードの修正はユーザーの確認後に行う** — 上記1・2をユーザーが確認・承認するまで、コード（実装ファイル）の編集には着手しない。

## 何が「設計の変更」に当たるか

- データモデルの変更（Notionのプロパティ追加・削除、型定義の変更など）
- 処理フロー・パイプラインの段数や責務分割の変更（例: AI生成を1段階から5段階に分けるような変更）
- 新しい外部サービス・認証方式の導入
- ディレクトリ構成やモジュール境界の変更

単純なバグ修正、タイポ修正、既存設計の範囲内でのリファクタリングはこのプロセスの対象外（通常のフローでよい）。

## 実務上の進め方

このプロジェクトでは Claude Code の Plan Mode（`EnterPlanMode` → 設計書をplanファイルに書く → `ExitPlanMode` でユーザー承認）がこのプロセスに対応する。設計変更の依頼を受けたら、まずPlan Modeに入り、必要なら `AskUserQuestion` で論点を確認し、承認が出てから実装に移る。
